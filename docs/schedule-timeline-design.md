# Schedule Timeline Design

## Overview

This document defines the design policy for schedule timeline output.

It covers the following two artifacts.

- JSON data generated by the exporter
- Static HTML template that renders the JSON

## Design Policy

The schedule timeline must be designed so that the exporter determines the displayed values and the HTML template only renders them.

The responsibilities are defined as follows.

- The exporter must include window metadata, slot display labels, run display timestamps, and next invocation display timestamps in the JSON output.
- The HTML template must render the JSON values as-is.
- The HTML template must not perform timezone conversion for display.
- The HTML template must not perform display-oriented time string formatting.
- The HTML template may keep only rendering logic such as hover handling, filtering, and DOM construction.

## Rationale

JavaScript Date in the browser interprets and displays timestamps in the viewer's local timezone unless the timezone is handled explicitly.

For that reason, the following logic must not live in the HTML template.

- Slot display label generation
- Run start and end display label generation
- Next invocation display label generation

If those values are generated in the browser, the same JSON can produce different visible results depending on the viewer environment.

Example:

- The JSON is based on UTC.
- The viewer browser is running in JST.
- Slot labels are displayed in JST.
- The run list still shows UTC strings.

In that state, the timeline axis, tooltip, and run list no longer share the same time basis.

## Output Contract

The exporter must emit display-ready values into the JSON output.

Representative fields are listed below.

- `window.hour_labels`
- `window.slot_labels`
- `schedules[].schedule_expression_timezone_label`
- `schedules[].next_invocation_label`
- `schedules[].runs[].start_label`
- `schedules[].runs[].end_label`

These values must be generated according to the timezone selected by the CLI.

## Template Constraints

The following behaviors are prohibited in the HTML template.

- Display timestamp generation using `getHours()` or `getMinutes()`
- Display timestamp generation using `Intl.DateTimeFormat`
- Display string generation based on the browser local timezone

Only the following behaviors are allowed in the HTML template.

- Rendering display strings from JSON into the DOM
- Run and slot overlap checks
- Tooltip visibility control
- Filter and sort UI control

## Tooltip Timezone Metadata

The tooltip must always show both schedule expression timezone and rendering timezone.

- `expression timezone`: value from `schedules[].schedule_expression_timezone_label`
- `display timezone`: value from `timezone`

Example:

- Expression timezone: `Asia/Tokyo (UTC+09:00)`
- Display timezone: `UTC`

This prevents confusion when schedule definitions and rendered timeline use different timezones.

## ECS Historical Run Backfill

ECS `ListTasks` and `DescribeTasks` cannot reliably provide previous-day stopped task history.

To keep timeline consistency for lookback windows such as previous full day, the collector supplements ECS runs with CloudTrail management events.

- CloudTrail event source: `ecs.amazonaws.com`
- Event name: `RunTask`
- Merge policy: detailed ECS task records are preferred when both ECS API and CloudTrail entries exist for the same task

## Implementation Notes

In the current implementation, display labels are generated by the Go exporter.

- Window label generation: `internal/exporter/cron.go`
- Run and next invocation label generation: `internal/exporter/cron.go`
- HTML rendering: `internal/exporter/html_template.html`

With this structure, generated artifacts produce the same visible output regardless of the viewing environment.

## Future Direction

If stricter separation is needed later, the exporter can also precompute the index information required for run and slot overlap checks and remove Date-based calculations from the HTML template entirely.

In that model, the HTML template becomes a fully render-only component.
